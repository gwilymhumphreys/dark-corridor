## The approach (keep doing this)

- **One file per system.** `docs/systems/<system>.md`. Cross-system edges live **once** in `architecture.md` → *Interface contracts (boundary hub)*; PRDs link there, never duplicate edges.
- **Incremental.** Foundation + combat-adjacent are PRD'd. **Run-structure (`Run manager` / `Encounter` / `Draft`) is now prototype-scope** — the prototype target is a **playable itch.io build** (see Next steps), so it needs the run controller, encounters/events, fights, and localization. **Meta / characters stay deferred** until that build exists (per architecture's "don't write all PRDs up front"). The separate files are *so* we can add layers one at a time.
- **PRD template:** Purpose (is / is-not) → Boundaries (link to hub) → Model & rules → Prototype scope → Open/deferred → Dependencies. Numbers point to source files, never baked into prose. Open questions are **flagged, not invented**.
- **Think-first, then review.** Before writing a PRD, reason through ambiguities, surface *genuine* forks to the user (not conventional defaults), resolve, then write. After writing, consistency-review against the other docs.
- **House style:** sister projects `../battledraft`, `../dogmage` (shared systems, the `Manager` naming). `../a-machine` (= AMTKAG, the dev's shipping game) is the model for the autotest + `tune` patterns.

## Key decisions + why

*(Naming evolved — don't resurrect old terms: `Core`/`Main` → `Timekeeper`; `…controller` → `…manager`; `StatusEngine` → `StatusManager`; `effect`/`pending effect` → `Delivery` + `payload`.)*

1. **`Timekeeper` (was Core/Main).** The combat time system is **combat-scoped** (resets each fight, no meaning outside) — "Core" wrongly implied global/persistent.
2. **`Manager` naming** (house style). Instanced → `class_name FooManager`; autoload → `class_name FooManagerAutoload` registered `Foo`. (`StatusEngine`→`StatusManager`; the two "controllers" → `Run manager` / `Combat manager`.)
3. **Testbed renamed** `main` → `corridor_testbed` (it's a renderer testbed, not the game; frees `main`).
4. **`Combat manager` introduced** — the per-fight orchestrator that nothing owned: the live fight (actor pair + left-to-right ordering), win/loss, runtime targeting authority, the trigger event bus, the component registry, **and the `Timekeeper`'s lifecycle**. The `Run manager` does **not** touch the Timekeeper (the user was explicit) — it hands a fight to the Combat manager and waits for the result.
5. **`StatusManager` = stateless-rules autoload.** Status *instances* live on their **targets** (Actor / Item), not in the manager — purely so the autoload holds no per-fight state. (Statuses themselves are combat-scoped, never run-persistent — **#26**; instances-on-targets is *not* a persistence mechanism.) **Block** = an actor-targeted **absorb-status**; `Actor.take_damage` runs incoming damage through the target's modifier-statuses via the StatusManager (block absorbs before HP; precise amplifier/absorber order is open — StatusManager PRD). Block **persists until consumed** (no decay) and absorbs all damage except `unblockable` payloads — a **per-effect** flag (varies by DoT), not a blanket DoT rule.
6. **Relics & potions = run-level** (player run-state `{ actor, relics, potions, … }`), **not** Actor-owned — keeps the Actor a pure, symmetric combatant (HP + board + statuses).
7. **Symmetric `Actor`; `Enemy` is not a class** — an Actor built from an authored *enemy definition*. Enemy items are a **content category** (per-enemy attack item + small shared utility pool), mechanically just `Item`s.
8. **`Delivery` + `payload`.** What an item fires = a **Delivery** (the in-flight carrier: `payload` + resolved target + travel-Ticker); its content = a **payload** `(kind, value)`. Retired "effect" as the combat-resolution word (overloaded). The item declares a **relative target-shape** (self / opponent-leftmost / all-opponents, + enemy-item shapes — see #14); the **Combat manager resolves shape→target and spawns the Delivery** — so the Item never depends *up* on the Combat manager.
9. **Fixed timestep (the big one).** Combat runs in fixed `STEP`s. The **`Timekeeper` is slimmed to the combat clock**: `sim_time` (stepped, for logic), `render_time()` (continuous, for the smooth VFX wall), the one speed `dial`, and `steps_due()` (real time × dial → whole steps, `MAX_STEPS` cap + backlog-drop) + `advance()`. The **`Combat manager` owns the component registry, advances every component each `sim_step()`, and runs the `_physics_process` loop**. The **dial = how many fixed steps run per real second** (not a per-step delta). This gives determinism + **bit-reproducible autotest** (run K steps); `render_time` keeps motion smooth between steps; Godot's physics loop handles the fixed-`delta` cadence + catch-up cap; **autotest drives `sim_step()` directly** (no real time).
10. **Item types:** every item is **active** (ticks); a "trigger" is an **additional event-push on the same accumulator** (the item still ticks — not a separate type); **no passive item type** (passive/always-on effects are **statuses**).
11. **`Save` = push, not pull.** The `Run manager` hands `Save` a snapshot on encounter entry; on launch the `Game manager` reads it back and the `Run manager` rehydrates — `Save` reads no live state. Auto-save on encounter entry, single run slot, **no migration** (`CLAUDE.md`). Snapshot = run-persistent state only (HP, board+enchants, relics, potions, position, **RNG** so resume is deterministic / not save-scummable); combat state is ephemeral, never saved.
12. **Triggers = accrual-only** (combat_model.md's Bazaar lesson — loop-proof). The **event bus lives in the Combat manager**; a push fires on the *next* step → **one link per step** (no in-step recursion).
13. **Autotest scaffolding.** The AI driver is just **another input-intent source** (combat is auto, so it only makes draft / choice / potion decisions); `--speed` = the Timekeeper dial; headless skips the output layer; determinism = seeded RNG. **The decision AI and the `tune` skill internals are deferred.**
14. **Item-targeting (enemy items) = random by default.** Effects that hit an opponent's *items* (silence, item-debuffs) get two shapes — `opponent-item-random` (one) and `all-opponent-items` (all); single-item selection is **random via the seeded combat RNG** (fights stay bit-reproducible). A deliberate exception to the actor rule ("leftmost, never random," for player predictability) — provisional, may become a rule after testing.
15. **`Run manager` split into `Game manager` + `Run manager`.** The old "Run manager" had grown to own session, run, *and* save concerns. Split by lifetime: **`Game manager`** (autoload `Game`, session-lifetime) owns the **game-state machine** (title → run → death → meta), the **run lifecycle** (create / resume / end the run), and the **save-lifecycle** calls (`read` on launch, `clear` on death/win, meta-save) + global "call-from-anywhere" hooks. **`Run manager`** (instanced per run) owns the **map + encounter sequencing + player run-state + HP-economy**, and **writes** the per-encounter snapshot. Completes the per-lifetime orchestrator pattern — **Game (session) → Run (descent) → Combat (fight)**. *`Game` is an autoload by deliberate exception:* a session-lifetime singleton holding session state matches the autoload lifetime (nothing to keep "fresh"); the guardrail is that **per-run state stays in the instanced `Run manager`** (`Game` holds only a reference, null between runs).
16. **`Encounter` = the per-beat tier; the choice layer's *assembly* is the `Run manager`'s.** An `Encounter` is one resolved beat (fight / event / rest), instanced per beat by the `Run manager`; a fight `Encounter` creates the `Combat manager`. Tier: **Game → Run → Encounter → Combat**. The 2–3 candidate options at a choice point are **drawn by the `Run manager`** (a pool-draw under act constraints + RNG — it owns the map); UI presents; the picked candidate becomes the live `Encounter`. The Encounter is the resolved unit, not the selector; tier-2 within-choices (an event's binary, an elite's engage-by-picking) are the Encounter's own resolution. Rewards: the `Encounter` reports its outcome + reward-kind up; the `Run manager` fulfills it (drives `Draft` / grants the relic).
17. **`Draft` = a stateless reward-draw service.** `Draft` (autoload, like `StatusManager` / `Save`) answers "what candidates to offer" — slot composition (usually item; low chance enchant/potion), **depth-weighting** (rarity odds rise with depth), seeded from the **run RNG** (offer fixed per run-state — no save-scum). The **`Run manager` holds the pending offer and applies the pick** (board / potion / enchant-target / relic); `Draft` writes nothing. **Weighting is depth/rarity only — never the build/archetype** (the design's rejected hidden-weighting). **No skip** (no cap → taking one is always correct; the decision is which).
18. **The next encounter approaches from depth (created after the draft).** Rather than advance-then-spawn, the `Run manager` **creates the next `Encounter` right after the reward/draft** (spawning its enemy `Actor`s at the vanishing point) and **auto-saves** (encounter entry); the corridor advance then **animates the approach** — the enemies scale up from depth into full view, so the walk *is* the encounter arriving, not dead time. **Resolution begins on arrival** (a fight `Encounter` creates its `Combat manager` then). The approach is presentation (corridor renderer + UI); the logical beat is the resolution.
19. **Content layer (Relic / Enchantment / Consumable) → one combined PRD, prototype-scope.** The three were referenced across the loop but unspecced; they're **thin** (lean on `StatusManager` / `Combat manager` / `Item`), so one `content.md` covers all three rather than three files. A relic = a persistent run-state modifier (combat-start status / triggered Ticker / direct mod); an enchant = a one-per-item modifier hooking the host item's fire/resolve; a consumable = a manually-fired reserve (a Delivery on throw, no Ticker). Split the file later if one grows. All three are **`Draftable`s** alongside `Item` (drafted / inspected / tooltipped); Enchantment differs only in *application* — it attaches to a chosen item, not its own slot.
20. **The `Run manager` owns the run RNG; its full state persists through save/load.** A single seeded PRNG drives all run-level randomness (draft offers, encounter assembly). The snapshot saves the **full RNG state** (not just the seed), so a reloaded save reproduces the **same future outcomes every time** — deterministic resume, no save-scumming a bad draft by quit-reload. Per-fight combat randomness (random item-targeting, #14) draws from a **derived per-fight stream** (run seed + encounter index), so combat doesn't perturb the run stream and a re-entered fight replays identically.
21. **`Draftable` = a composition contract, not a parent class.** What a draft can offer (item / relic / consumable / enchant) shares a **definition-face** (`id / name / icon / rarity / category / tooltip`) by composition; `Draft` + inspection are **category-blind**, and **application dispatches on `category`** (the one switch, in the `Run manager`). The **runtime entities stay distinct types** (they share almost nothing at runtime — a parent class would be a near-empty fat base). Consistent with the Ticker (owned, not subclassed) and Status (data + type key). `category` is **open** — adding a kind is additive.
22. **Entity model is already ally-ready — keep it that way (deferred experiment).** Two families — **`Actor`** (player + enemy, unified by symmetry) and **`Draftable`** (#21) — make "drafting allies" *additive*, not a refactor: an ally = a `Draftable` (`category: ally`) whose application spawns an `Actor` on the player's **side**. The `Combat manager` is already side-relative + list-based (N-vs-M in the abstraction; only 1-vs-M exercised today); the open `category` absorbs the new kind. **Don't build it now** (prototype stays 1-vs-M); **don't over-unify** (no über-`Entity` base over `Actor` + `Draftable` — they share nothing at runtime). The only single-player assumption is run-state's singular `actor` + its snapshot — cheap to change later (no save migration). **Forward-compat rule:** treat each side as a *party* (a roster of 1 today); don't hardwire "exactly one player `Actor`." Same "side = roster" need as the deferred summoning item.

23. **Content definitions are authored in GDScript, not data files.** Item / enemy / status / encounter / relic / enchant / consumable definitions are typed `RefCounted` def objects (`ItemDef`, `EnemyDef`, …) carrying the `Draftable` header by composition, constructed in a **static per-kind catalog keyed by a string id** (amended 2026-06-07 from an int `enum Id` to string ids — `ItemCatalog.WEAPON == 'weapon'` — so content files can be added/removed without editing a central enum). No JSON / `.tres` / inspector — author everything in code (the Slay-the-Spire-2 approach; chosen over last project's JSON, to try). Localization is preserved: source English on the def, displayed via `tr(def.name)`, with the catalog files added to the POT extraction list. Resolves the item / status / enemy / encounter / content data-format opens. *(The `Save` run-snapshot serialization format is separate and still open — it stores ids + runtime values, not defs.)* *(Organization, 2026-06-07: each kind's defs + catalog live in a grouped subdir — `src/content/{items,enemies,relics,consumables,enchants,encounters,statuses,characters}/`.)*
24. **Deterministic component order = a monotonic `seq_id`.** The `Combat manager` stamps each registered component (item Ticker / status / Delivery) an incrementing `seq_id` at registration and advances its registry in ascending `seq_id`. Registration order is itself deterministic (board order at fight start, then the deterministic fire order each step), so the sweep is bit-reproducible. Simplest thing that works — resolves the Combat-manager *within-step component order* open. *(Phase 1 realization, 2026-06-05: the sweep is built as fixed type-ordered passes — item cooldowns → statuses → Delivery travel, each in insertion order — which delivers the same deterministic, bit-reproducible order. A literal `seq_id` counter field is deferred until components of different types must interleave by registration time, which Phase 1 never requires.)*
25. **Scene tree = a logic tree (under the `Game` autoload) + a separate presentation tree (the main scene).** Combat logic is plain `RefCounted` (`Actor` / `Item` / `Status` / `Delivery` / `Timekeeper` / event bus); only the three orchestrators (`Run manager` / `Encounter` / `Combat manager`) are `Node`s, and **only the `Combat manager` runs `_physics_process`** (the one tick). The presentation tree (`main.tscn` → `ScreenHolder` → title/run/death screens; the run screen composing the corridor panel + a swappable `CombatView` + overlays) only *reads* the logic and emits intents — so a headless autotest just doesn't mount it. The **framed-vs-full-screen** open is isolated to the single swappable `CombatView` sub-scene. Full structure + directory layout: [`architecture.md` → Scene tree & node model](systems/architecture.md#scene-tree--node-model). New dirs `src/combat/` · `src/run/` · `src/content/` · `src/vfx/` · `src/scenes/screens/` · `src/scenes/combat/`; `Balance` consts in `src/data/balance.gd`; GUT in `addons/gut/` + `tests/`.

26. **Statuses are combat-scoped; run persistence is Relics + Enchantments (status-lifetime resolved, 2026-06-05).** Every status (poison, block, burn, freeze, silence, this-fight item buffs/debuffs) is **combat-scoped** — created during a fight, cleared at `CombatManager.teardown()`, **never saved**. Statuses are *dynamic* combat states (accrue / decay / consume / expire) with no coherent cross-fight meaning (you don't carry a half-consumed block pool or a ticking poison between fights) — the Slay-the-Spire model the design follows (powers reset each combat; relics persist). **Durable state lives one layer up:** a **Relic** (run-level, saved — and a relic may carry its own counter) or an **Enchantment** (permanent item modifier, saved on the board) holds the magnitude, and re-expresses it as a fresh combat-scoped status each fight where needed — **Stone Ward** already does exactly this (a saved relic that applies a combat-scoped block status at each fight start). Consequence: **the run snapshot never serializes status instances** — only ids + values (board+enchants, relics, potions, HP, position, RNG). This **corrects the earlier framing** in #5 + the StatusManager PRD that statuses-on-`Item`s were a way to let item-buffs *outlive* a fight — they don't; that's the Enchantment's job. Instances live on their targets only to keep `StatusManager` **stateless**. Resolves the Save PRD's open "which statuses are run-persistent vs combat-scoped" → **none are**.

27. **Per-character item pools (departs from the single-shared-pool plan, 2026-06-07).** Each character draws from its **own** item pool, not one shared pool across all characters (which `design.md` originally specced). **Why:** a 1-of-3 draft with **no hidden weighting** (rejected — [design](design/game_design.md) / [draft](systems/draft.md)) can't be both *wide* and *connective* from one pool — few directions → drafts connect but the game is narrow; many directions → range but drafts stop connecting. Per-character pools dissolve the tradeoff by putting **focus per-run** (one character's few coherent directions) and **range per-game** (N× across the roster); pools can be **smaller and deeper** per character (unique synergies that wouldn't fit alongside everyone else's). **Hybrid rejected** (a shared neutral / Slay-the-Spire "colorless" commons layer + per-character anchors): our commons aren't thematically null like Slay the Spire's — a character's spore appliers are *also* its damage / block, so the "answer combat questions before synergy" need is met inside its own pool; a neutral layer would fight identity for no scope saving worth the cost. *(Refinement 2026-06-07: it's the colorless **layer** — a default tier characters depend on — that's rejected; individual **colorless items** are **not** banned and may still exist where they earn it (character-agnostic utility, or an item that fits several pools), just never as a workhorse-commons crutch.)* **Scope — split only the layer that needs depth:** **item pools** split per character; **enemies stay a shared pool**, the **reward-relic pool stays shared**, and per-character *everything* (relics / enemies / bespoke commons) is the **Battledraft scope trap** to avoid. *(The unique build-defining **starting relic** per character is unchanged — that's one bespoke relic, not a split pool.)* **Build impact — none to systems:** `Draft` already pulls from "the pool" and is category-blind (#21 — the `Draftable` contract); scoping the pool to the active character is a **content-organization** change. The only wiring: the `Run manager` hands the *character's* pool to `Draft.draw` instead of one global pool. **Watch:** directions-per-character is a starting peg, not a number (too few forces drafts; too many re-dilutes inside one character) — resolve in prototype. Full rationale: [`design/per_character_pools.md`](design/per_character_pools.md). **BUILT 2026-06-07:** a minimal `CharacterDef` + `CharacterCatalog` (a placeholder default character ports the prototype seed; the full Characters system stays deferred); `RunManager._draft_pool()` hands `character.item_pool + ColorlessPool.ITEMS` to `Draft` (the shared colorless pool is empty by default — the exception that earns it). 217 GUT tests green.

28. **Shared status baseline — Vulnerable + Weak available to every character (2026-06-08).** A character can apply a small **shared baseline** of generic statuses (Vulnerable, Weak to start) — the Slay-the-Spire model. **Why:** it lets a character's *bespoke* statuses stay **few and deep**. The routine debuffs are covered by the baseline, so the **Spore Druid** runs **one signature spore counter** — **Spores** (the stacked Mass fuel, doing nothing on its own for now — open to change) + 1–2 minor spores instead of a wide bestiary that just shadows engine statuses; a "cares-about-distinct-statuses" card then draws variety from baseline + bespoke without authoring many. **Not a reopening of #27:** this is a shared *status* vocabulary, **not** a shared *item* layer — the **appliers** still live in each character's own pool (#27 rejected a colorless commons *item* tier, not shared statuses). Vulnerable/Weak are already-built statuses (the stat-status seam); acquisition tunes per source (the design's asymmetric-acquisition note). Spore Druid specifics + the main-spore design: [`design/spore_druid.md`](design/spore_druid.md).

29. **Statuses are polymorphic `StatusEffect` classes, not a central rulebook (2026-06-10).** Replaces the original model (one `StatusManager` switching on `StatusDef.Shape`, statuses = data keyed by an int `Type` enum) with **a class per status** (the Slay-the-Spire `AbstractPower` model): each `StatusEffect` subclass owns its state *and* behaviour via overridable hooks (`modify_outgoing` / `modify_incoming` / `absorb` / `on_step` / `gates_fire` / `causes_evasion` / `is_fuel` …, all no-op/identity by default), extending an intermediate base (`TimedStatus` / `PeriodicStatus` / `PoolStatus`) or `StatusEffect` directly. **Why:** "many statuses, wildly different effects" bloats a central switch; a class-per-status keeps each effect in its own ~6-line file and lets the engine **stop naming statuses** (`type == BLOCK` → `status.absorb(...)`, the fuel check → `status.is_fuel()`). `StatusManager` demotes to a **stateless facade** that loops a target's `statuses` and calls hooks (preserving deterministic order #24 + amplify-before-absorb #6). **Decisions:** statuses become **string-id content** (#23 extended; the `Type` enum removed, `StatusDef`/`StatusCatalog` deleted, registered in `StatusRegistry`); hooks receive **`(target, ctx)`** so a status stores **no target ref** (keeps the no-RefCounted-cycle invariant from #5/#26); **duration rides the application** (`ItemEffect.duration` → the instance — killing the global-duration coupling) and **reapply stacks by default** (`TimedStatus` extends its timer). Costs nothing in saves (statuses combat-scoped, #26). **Supersedes** the rulebook framing in #5 + the StatusManager PRD; #25's list of logic classes now reads `StatusEffect` for `Status`. Model + hook interface: [`status_manager.md`](systems/status_manager.md). 251 GUT tests green; full-descent autotest wins.

30. **Gate-freeze · own-side trigger default · potion exemption (review fixes, ratified 2026-06-13).** Three combat-semantics calls from the full codebase review. **(a) A gated item's cooldown FREEZES:** while a gate status (silence) sits on an item, the Combat manager skips its accrual entirely — previously the accumulator kept filling and the gate lifting machine-gunned the backlog (a 10s silence on a 2s item released ~5 instant fires, making silence a damage *amplifier*). Now silence deletes that item's time; the first fire lands one full cooldown after the lift. (Considered and rejected: cap-at-one-pent-up-fire; keep-the-burst.) **(b) Trigger source defaults to OWN SIDE:** the event bus carries source identity, and an item's trigger subscription reads "when MY side does X" unless the def opts out (`source_filter`) — the Slay-the-Spire reading and the StatusManager PRD's canonical example; previously Sporeward-Buckler-style triggers charged off *anyone's* applications. **(c) Thrown consumables are exempt from the thrower's combat modifiers:** Weak doesn't scale a potion down, Blind can't whiff a throw — potions are the reserve, not the engine, so debuffs that degrade the board don't degrade the panic button. A deliberate, documented asymmetry with the item fire pipeline. Homes: [`item.md`](systems/item.md) (a), [`combat_manager.md`](systems/combat_manager.md) (b), [`status_manager.md`](systems/status_manager.md) + [`content.md`](systems/content.md) (c).

31. **Combat item tooltip = a cluster fed by a point-poll; per-keyword info via Godot built-in custom tooltips (gen 3, built 2026-06-19).** Hovering a board `Item` shows a **cluster** (a main panel + an always-shown keyword **column**) beside it; the column's cards are the at-a-glance meaning, and each body keyword **chip** *additionally* carries a Godot **built-in custom tooltip** (`_make_custom_tooltip` returns a **frameless** `keyword_card`; the theme's `TooltipPanel` is styled opaque as the only frame — the double-panel contract) so sub-tooltip positioning is the engine's, not ours (no manual recursion). **Driving:** a **point-poll** off the run screen's existing slow-mo hover (`view.update_inspection(mouse)` while FIGHTING + not paused), not per-cell signals — one hover paradigm, and the floating cluster's hide-bridge needs a per-frame rect check regardless. **Positioning:** default side (LEFT — the player board hugs the right edge), screen-half flip as the fallback, then clamp. **Values are read-only** — a pure `Item.display_value` / `base_value` (enchant mult + the outgoing stat-status seam; **never** `fire()`, which mutates), with a changed-value highlight; consume-scaling deferred (needs a non-mutating stack peek). **Keywords are catalog-gated** (`KeywordCatalog`): a status pulls its card from its `StatusEffect` subclass (one home), a mechanic (`kw:*`) appears only if authored — that absence is the gate. **The mouse-bridge stays** (one level: item↔cluster, so the cursor can reach the chips); the recursive bridge is dropped (Godot owns the keyword-tip lifetime). Scope v1: board items only (potions/draft are a follow-on). Full as-built: [`tooltips.md`](systems/tooltips.md); plan + rationale: [`plans/tooltip_system.md`](plans/tooltip_system.md). The **copy + colour treatment is the owner's**, scaffolded as marked placeholders (status `desc_key`s, mechanic descriptions, item `description_key`s, line templates, the highlight/rarity tint).

## Open / deferred (each has a home)

- Timescale override **replace-vs-multiply — resolved → replace** (absolute slow-mo, independent of the ×1/×2/×3 battle-speed dial; the dial is a `Game` session preference applied to each fight's Timekeeper base scale). timekeeper.md / combat_model.md.
- **Within-step component order — resolved (#24):** deterministic registration order; built in Phase 1 as fixed type-ordered passes (item cooldowns → statuses → Deliveries), with a literal `seq_id` counter deferred until cross-type interleaving is needed.
- **Stat-statuses** (strength/weak/vulnerable) → the two damage-modifier seams are now **BUILT** (as `StatusEffect` hooks, #29): `modify_outgoing` (applied at fire time in `Item._resolve_effect`) + the incoming **amplifier** stage (`modify_incoming`) in `resolve_incoming_damage` (before block), both **% multipliers** (cascade-safe — a flat *per-fire* modifier makes fast items strictly dominant). Placeholder `Weak` / `Vulnerable` statuses + a `Sundering Bolt` applier prove the path; the real stat-status **content** (numbers, per-stack variants, which damage types amplify) stays the owner's, authored as GD `StatusEffect` classes ([design](design/game_design.md)).
- **Per-effect stack/decrement** + block tuning → content.
- **Mid-fight roster changes (summoning) — BUILT (Cap 3, 2026-06-08).** The Combat manager is now a **two-roster** model (no longer a fixed pair): `add_actor` spawns a combat-scoped summon on either side, `Delivery.Kind.SUMMON` lands it, scope-aware teardown dissolves enemies + player tokens but keeps the run-scoped player + allies.
