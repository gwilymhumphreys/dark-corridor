# Dark Corridor — Build Log (chronological record)

> The dated record of what got built, in order — moved out of the decision log
> (which now holds decisions only). **Current** status lives in
> [`../handoff.md`](../handoff.md); this file is the history. The per-phase plans
> in this directory are the detailed step-by-steps each entry followed.

- **Phase 1 — the combat spine — is built and green.** The build-prep decisions (#23–25), then Steps 1–5 of [`phase1_plan.md`](phase1_plan.md): `Timekeeper` · `Actor` · `StatusManager` · `Item` · `CombatManager` + a minimal **opaque** VFX wall + the `combat_sandbox` host. **45 GUT tests pass headless on 4.6; the feel gate passed.** Code now lives in `src/combat/`, `src/content/`, `src/autoloads/status_manager.gd`, `src/data/balance.gd` (placeholder tuning), `src/vfx/`, `src/scenes/combat/`, `tests/`. GUT is vendored in `addons/gut/`.

- **Phase 2 — autotest scaffolding — is built and green** (2026-06-05): the Mode/Driver/Logger trio + `AutoTestStuckDetector` in `src/autotest/` (dedicated `autotest.tscn`) drive ONE deterministic headless fight to a verdict via `CombatManager.sim_step()` — flags, forced nosave/notutorial, game-time `--timeout` + `--wall-timeout` + stuck detection, damage-by-family logging + a markdown report, exit `0` (resolved) / `1` (stuck/timeout). `--seed` / `--speed` are plumbed-but-inert until the run RNG (Phase 3). A Phase-1 review pass first hardened the spine (heal guard, Delivery pruning, seq_id doc).

- **Phase 3 — the run loop — is built and green** (2026-06-05, [`phase3_plan.md`](phase3_plan.md)): `Save` (autoload, JSON snapshot, atomic, no migration) + minimal `Relic` (combat-start status) + `Draft` (autoload, seeded 1-of-3) + `Encounter` (fight/rest, in `src/run/`) + `RunManager` (map, run-state, sequencing, HP-economy, snapshot/rehydrate, in `src/run/`) + `GameManagerAutoload` (`Game`: phase machine + run lifecycle). The autotest now drives a whole headless descent (`run_full`) — draft → fight → advance → win, deterministic by `--seed` (now live), with quit/resume. New autoloads `Save` / `Draft` / `Game`; new dir `src/run/`.

- **Fast-follow done (2026-06-06):** the **enchant** (Whetstone, scale-a-value on the board) + **consumable** (Healing Draught, thrown self-heal via `CombatManager.throw_consumable`) categories — so all three content categories prove their path end-to-end in the run + autotest. Also resolved **decision #26** (statuses are combat-scoped; persistence = relics/enchants) and fixed the **`Actor`↔`Item` RefCounted leak** (`Actor.dissolve()` at discard) + made status advancement uniform across actors and items. **112 GUT tests pass headless on 4.6.**

- **Phase 4 — real UI / the run screen — is built and green** (2026-06-06, [`phase4_plan.md`](phase4_plan.md)): the presentation tree (`main.tscn` → `MainController` → title / run / outcome screens) + the **real-time seam** (the run screen drives `CombatManager.tick` each physics frame; the logic tree stays *out* of the scene tree — the autotest path is byte-identical) + the **framed** combat view (corridor + thorn-demon **occupant** top-right via `CorridorScaled.axis_scale`, player left, board strips / HP / potions, the VFX wall) + slow-mo-on-hover + the run-screen **FSM** (approach → fight → draft → advance → win/death) + the **draft overlay** + the **map strip** + the **corridor approach** (the enemy scales from depth). New dirs `src/scenes/screens/` + `src/scenes/combat/` (the view); `main_scene` flipped to `main.tscn`. Full doc: [`systems/run_screen.md`](../systems/run_screen.md). Deviations: the `RunManager.advancing` signal was unneeded (the run screen orchestrates the advance + approach directly, keeping `RunManager` presentation-free); one parameterized `outcome_screen.tscn` instead of two. **126 GUT tests pass headless on 4.6.** Two follow-ups done (128 tests): the **potion-throw UI** (clickable slots → `RunManager.throw_potion`) + the **localization POT pipeline** (`tools/extract_pot.gd` → `locale/`, registered in `project.godot`; [`localization.md`](../systems/localization.md)).

- **Phase 5 — the `tune` machinery — is built** (2026-06-06, [`phase5_plan.md`](phase5_plan.md), 139 tests): seeded board-aware **draft strategies** (`--strategy` live) + a **per-encounter / per-item report** (durations vs window, HP attrition, item fires + damage + trap-pick flags) — the two things the `tune` workflow needs.

- **Non-content backlog item — settings/pause + battle-speed — built** (2026-06-06, 153 tests): the **×1/×2/×3 battle-speed dial** (a `Game` session preference + an always-visible HUD toggle, applied to each fight's `Timekeeper` base scale) + in-run **pause** (a run-screen gate, **not** a `Game` phase) with a pause menu (Resume / Quit-to-menu, the save preserved so Title's Resume re-enters the beat); also **resolved the timescale replace-vs-multiply open → replace** (absolute slow-mo, independent of the dial).

- **Non-content backlog item — DoT per-applier attribution — done** (2026-06-06, 158 tests): the tune report's damage-by-family + item-contribution table now credit poison-tick damage to the **item that applied it** (a pre-step status snapshot reads `Status.source`), so Venom Fang shows its real poison damage instead of a generic "Poison" lump reading 0 — sharpening what `tune` reads (in `src/autotest/`).

- **Run structure + reward routing + status-build engine seams — built (2026-06-07):** the **multi-act run** (`RunMap` = 3 acts × 15 beats; act-end bosses, a fixed midpoint relic + a per-act rest, the rest **CHOICE** beats) with the **choice layer** (`has_pending_choice` / `pending_choice` / `pick_path`, drawing 2-3 seeded candidates), a **two-tier choice UI** (`choice_overlay`) + a placeholder **Event** type (`event_overlay` / `pick_event_option`, prose + binary outcome), the run-screen FSM gaining CHOOSING / EVENTING states, and the **HP economy** (between-act full heal, per-act rest) (#1, #16); **reward routing** — a relic on the **RELIC** reward + a relic-and-draft on the new **ELITE** reward, drawn from `RelicCatalog.REWARD_POOL` on the run RNG (deterministic / resume-stable), plus a relic `MAX_HP_BONUS` shape (#2); **per-fight RNG** (`_combat_seed_for` = run seed + encounter index) + a random-targeting item exercising the derived per-fight stream (#14, #20); the **spore-engine seams** Cap 1+2 ([`spore_engine.md`](../systems/spore_engine.md)) — `StatusManager.consume(target, type, amount)` stack-spending (self-fuel in the Item pipeline, opponent-fuel in the Combat-manager spawn path) + an **evasion** fizzle seam (`causes_evasion` → `Delivery.evaded`, a whiff with a reason, distinct from silence/`gate` = inert); and **mid-fight roster changes** Cap 3 — the `CombatManager` **two-roster** model (player side = run actor + run-scoped `allies` + combat-scoped `_player_tokens`; enemy side = `enemies`) with `add_actor` (combat-scoped summon), `register_ally` (run-scoped mid-fight ally), `Delivery.Kind.SUMMON`, and scope-aware teardown — so **allies support both run and combat scope** (#22). All placeholder content; the mechanisms are wired.

- **Content scaffolding built (2026-06-07, 217 tests):** content ids are now **string ids** (#23, amended from int enum), `src/content/` is reorganized into **kind-grouped subdirs**, a **minimal character system** (`CharacterDef`/`CharacterCatalog`) drives the start kit + a **per-character draft pool** (#27), and a shared **colorless pool** (empty by default) is appended at draw.

- **Multi-actor combat view + pause-anytime + mid-fight ally wiring — built (2026-06-08, 223 tests):** the framed `CombatView` now renders a **board strip per actor** (every enemy in the right column, the player's prominent board bottom-left, each run-scoped ally / combat-scoped summon token in a column beside it — reading the CM rosters each frame, so mid-fight summons appear as they spawn; `actor_pos` resolves each to its own strip) (#4); **pause is available at any point** in a live run (a run-screen gate, the menu's full-rect catcher blocking the board underneath) (#8); `RunManager.add_ally` registers a run-scoped ally into the **live** fight mid-combat (#6); plus a review pass — catalog `get_def` id-guards (an unknown id `push_error`s instead of a null-deref crash), a periodic-status-on-item guard, and `_payload_from_effect` / `throw_consumable` summon+consume field parity.

- **Ally acquisition + character-select + settings + combat-view relayout (2026-06-09):** a recruit Event grants a run-scoped ally (4-slot cap); a **character-select** screen + a **settings** screen (audio-volume `Prefs` persisted to `user://`); the combat view relaid out **corridor-forward** (corridor large top-left, an `enemy_hud` pinned above each occupant sprite, player portrait + HP centre-bottom, its board a column down the right edge, allies in flanking slots) + a **compact windowed map**.

- **Status system refactored + first Spore Druid weapons (2026-06-10, #29):** statuses are now polymorphic `StatusEffect` classes (string-id, per-application duration, one file per status) behind a stateless facade — `StatusDef`/`StatusCatalog`/the `Type` enum removed; the beat model went CHOICE→ROLL (beats auto-roll combat/event, the choice layer dormant); the Spore Druid gained its first common attacks (a DPS = cooldown + 3 curve) + Wilt Frond; and every gameplay/UI colour + shared draw-const was centralized into `Colours` / `Consts`. **251 GUT tests green.**

- **Next: the deferred content push** — scale the per-character item / enemy / encounter pools, the boss signature mechanics + the Spore Druid spore content, then a real `tune` pass.
