## Key decisions + why

*(Naming evolved â€” don't resurrect old terms: `Core`/`Main` â†’ `Timekeeper`; `â€¦controller` â†’ `â€¦manager`; `StatusEngine` â†’ `StatusManager`; `effect`/`pending effect` â†’ `Delivery` + `payload`.)*

1. **`Timekeeper` (was Core/Main).** The combat time system is **combat-scoped** (resets each fight, no meaning outside) â€” "Core" wrongly implied global/persistent.
2. **`Manager` naming** (house style). Instanced â†’ `class_name FooManager`; autoload â†’ `class_name FooManagerAutoload` registered `Foo`. (`StatusEngine`â†’`StatusManager`; the two "controllers" â†’ `Run manager` / `Combat manager`.)
3. **Testbed renamed** `main` â†’ `corridor_testbed` (it's a renderer testbed, not the game; frees `main`).
4. **`Combat manager` introduced** â€” the per-fight orchestrator that nothing owned: the live fight (actor pair + left-to-right ordering), win/loss, runtime targeting authority, the trigger event bus, the component registry, **and the `Timekeeper`'s lifecycle**. The `Run manager` does **not** touch the Timekeeper (the user was explicit) â€” it hands a fight to the Combat manager and waits for the result.
5. **`StatusManager` = stateless-rules autoload.** Status *instances* live on their **targets** (Actor / Item), not in the manager â€” this is what lets persistent item-buffs outlive a fight, and it keeps an autoload from holding per-fight state. **Block** = an actor-targeted **absorb-status**; `Actor.take_damage` runs incoming damage through the target's modifier-statuses via the StatusManager (block absorbs before HP; precise amplifier/absorber order is open â€” StatusManager PRD). Block **persists until consumed** (no decay) and absorbs all damage except `unblockable` payloads â€” a **per-effect** flag (varies by DoT), not a blanket DoT rule.
6. **Relics & potions = run-level** (player run-state `{ actor, relics, potions, â€¦ }`), **not** Actor-owned â€” keeps the Actor a pure, symmetric combatant (HP + board + statuses).
7. **Symmetric `Actor`; `Enemy` is not a class** â€” an Actor built from an authored *enemy definition*. Enemy items are a **content category** (per-enemy attack item + small shared utility pool), mechanically just `Item`s.
8. **`Delivery` + `payload`.** What an item fires = a **Delivery** (the in-flight carrier: `payload` + resolved target + travel-Ticker); its content = a **payload** `(kind, value)`. Retired "effect" as the combat-resolution word (overloaded). The item declares a **relative target-shape** (self / opponent-leftmost / all-opponents, + enemy-item shapes â€” see #14); the **Combat manager resolves shapeâ†’target and spawns the Delivery** â€” so the Item never depends *up* on the Combat manager.
9. **Fixed timestep (the big one).** Combat runs in fixed `STEP`s. The **`Timekeeper` is slimmed to the combat clock**: `sim_time` (stepped, for logic), `render_time()` (continuous, for the smooth VFX wall), the one speed `dial`, and `steps_due()` (real time Ă— dial â†’ whole steps, `MAX_STEPS` cap + backlog-drop) + `advance()`. The **`Combat manager` owns the component registry, advances every component each `sim_step()`, and runs the `_physics_process` loop**. The **dial = how many fixed steps run per real second** (not a per-step delta). This gives determinism + **bit-reproducible autotest** (run K steps); `render_time` keeps motion smooth between steps; Godot's physics loop handles the fixed-`delta` cadence + catch-up cap; **autotest drives `sim_step()` directly** (no real time).
10. **Item types:** every item is **active** (ticks); a "trigger" is an **additional event-push on the same accumulator** (the item still ticks â€” not a separate type); **no passive item type** (passive/always-on effects are **statuses**).
11. **`Save` = push, not pull.** The `Run manager` hands `Save` a snapshot on encounter entry; on launch the `Game manager` reads it back and the `Run manager` rehydrates â€” `Save` reads no live state. Auto-save on encounter entry, single run slot, **no migration** (`CLAUDE.md`). Snapshot = run-persistent state only (HP, board+enchants, relics, potions, position, **RNG** so resume is deterministic / not save-scummable); combat state is ephemeral, never saved.
12. **Triggers = accrual-only** (combat_prd's Bazaar lesson â€” loop-proof). The **event bus lives in the Combat manager**; a push fires on the *next* step â†’ **one link per step** (no in-step recursion).
13. **Autotest scaffolding.** The AI driver is just **another input-intent source** (combat is auto, so it only makes draft / choice / potion decisions); `--speed` = the Timekeeper dial; headless skips the output layer; determinism = seeded RNG. **The decision AI and the `tune` skill internals are deferred.**
14. **Item-targeting (enemy items) = random by default.** Effects that hit an opponent's *items* (silence, item-debuffs) get two shapes â€” `opponent-item-random` (one) and `all-opponent-items` (all); single-item selection is **random via the seeded combat RNG** (fights stay bit-reproducible). A deliberate exception to the actor rule ("leftmost, never random," for player predictability) â€” provisional, may become a rule after testing.
15. **`Run manager` split into `Game manager` + `Run manager`.** The old "Run manager" had grown to own session, run, *and* save concerns. Split by lifetime: **`Game manager`** (autoload `Game`, session-lifetime) owns the **game-state machine** (title â†’ run â†’ death â†’ meta), the **run lifecycle** (create / resume / end the run), and the **save-lifecycle** calls (`read` on launch, `clear` on death/win, meta-save) + global "call-from-anywhere" hooks. **`Run manager`** (instanced per run) owns the **map + encounter sequencing + player run-state + HP-economy**, and **writes** the per-encounter snapshot. Completes the per-lifetime orchestrator pattern â€” **Game (session) â†’ Run (descent) â†’ Combat (fight)**. *`Game` is an autoload by deliberate exception:* a session-lifetime singleton holding session state matches the autoload lifetime (nothing to keep "fresh"); the guardrail is that **per-run state stays in the instanced `Run manager`** (`Game` holds only a reference, null between runs).
16. **`Encounter` = the per-beat tier; the choice layer's *assembly* is the `Run manager`'s.** An `Encounter` is one resolved beat (fight / event / rest), instanced per beat by the `Run manager`; a fight `Encounter` creates the `Combat manager`. Tier: **Game â†’ Run â†’ Encounter â†’ Combat**. The 2â€“3 candidate options at a choice point are **drawn by the `Run manager`** (a pool-draw under act constraints + RNG â€” it owns the map); UI presents; the picked candidate becomes the live `Encounter`. The Encounter is the resolved unit, not the selector; tier-2 within-choices (an event's binary, an elite's engage-by-picking) are the Encounter's own resolution. Rewards: the `Encounter` reports its outcome + reward-kind up; the `Run manager` fulfills it (drives `Draft` / grants the relic).
17. **`Draft` = a stateless reward-draw service.** `Draft` (autoload, like `StatusManager` / `Save`) answers "what candidates to offer" â€” slot composition (usually item; low chance enchant/potion), **depth-weighting** (rarity odds rise with depth), seeded from the **run RNG** (offer fixed per run-state â€” no save-scum). The **`Run manager` holds the pending offer and applies the pick** (board / potion / enchant-target / relic); `Draft` writes nothing. **Weighting is depth/rarity only â€” never the build/archetype** (the design's rejected hidden-weighting). **No skip** (no cap â†’ taking one is always correct; the decision is which).
18. **The next encounter approaches from depth (created after the draft).** Rather than advance-then-spawn, the `Run manager` **creates the next `Encounter` right after the reward/draft** (spawning its enemy `Actor`s at the vanishing point) and **auto-saves** (encounter entry); the corridor advance then **animates the approach** â€” the enemies scale up from depth into full view, so the walk *is* the encounter arriving, not dead time. **Resolution begins on arrival** (a fight `Encounter` creates its `Combat manager` then). The approach is presentation (corridor renderer + UI); the logical beat is the resolution.
19. **Content layer (Relic / Enchantment / Consumable) â†’ one combined PRD, prototype-scope.** The three were referenced across the loop but unspecced; they're **thin** (lean on `StatusManager` / `Combat manager` / `Item`), so one `content_prd.md` covers all three rather than three files. A relic = a persistent run-state modifier (combat-start status / triggered Ticker / direct mod); an enchant = a one-per-item modifier hooking the host item's fire/resolve; a consumable = a manually-fired reserve (a Delivery on throw, no Ticker). Split the file later if one grows. All three are **`Draftable`s** alongside `Item` (drafted / inspected / tooltipped); Enchantment differs only in *application* â€” it attaches to a chosen item, not its own slot.
20. **The `Run manager` owns the run RNG; its full state persists through save/load.** A single seeded PRNG drives all run-level randomness (draft offers, encounter assembly). The snapshot saves the **full RNG state** (not just the seed), so a reloaded save reproduces the **same future outcomes every time** â€” deterministic resume, no save-scumming a bad draft by quit-reload. Per-fight combat randomness (random item-targeting, #14) draws from a **derived per-fight stream** (run seed + encounter index), so combat doesn't perturb the run stream and a re-entered fight replays identically.
21. **`Draftable` = a composition contract, not a parent class.** What a draft can offer (item / relic / consumable / enchant) shares a **definition-face** (`id / name / icon / rarity / category / tooltip`) by composition; `Draft` + inspection are **category-blind**, and **application dispatches on `category`** (the one switch, in the `Run manager`). The **runtime entities stay distinct types** (they share almost nothing at runtime â€” a parent class would be a near-empty fat base). Consistent with the Ticker (owned, not subclassed) and Status (data + type key). `category` is **open** â€” adding a kind is additive.
22. **Entity model is already ally-ready â€” keep it that way (deferred experiment).** Two families â€” **`Actor`** (player + enemy, unified by symmetry) and **`Draftable`** (#21) â€” make "drafting allies" *additive*, not a refactor: an ally = a `Draftable` (`category: ally`) whose application spawns an `Actor` on the player's **side**. The `Combat manager` is already side-relative + list-based (N-vs-M in the abstraction; only 1-vs-M exercised today); the open `category` absorbs the new kind. **Don't build it now** (prototype stays 1-vs-M); **don't over-unify** (no ĂĽber-`Entity` base over `Actor` + `Draftable` â€” they share nothing at runtime). The only single-player assumption is run-state's singular `actor` + its snapshot â€” cheap to change later (no save migration). **Forward-compat rule:** treat each side as a *party* (a roster of 1 today); don't hardwire "exactly one player `Actor`." Same "side = roster" need as the deferred summoning item.

23. **Content definitions are authored in GDScript, not data files.** Item / enemy / status / encounter / relic / enchant / consumable definitions are typed `RefCounted` def objects (`ItemDef`, `EnemyDef`, â€¦) carrying the `Draftable` header by composition, constructed in a **static per-kind catalog keyed by an int id**. No JSON / `.tres` / inspector â€” author everything in code (the Slay-the-Spire-2 approach; chosen over last project's JSON, to try). Localization is preserved: source English on the def, displayed via `tr(def.name)`, with the catalog files added to the POT extraction list. Resolves the item / status / enemy / encounter / content data-format opens. *(The `Save` run-snapshot serialization format is separate and still open â€” it stores ids + runtime values, not defs.)*
24. **Deterministic component order = a monotonic `seq_id`.** The `Combat manager` stamps each registered component (item Ticker / status / Delivery) an incrementing `seq_id` at registration and advances its registry in ascending `seq_id`. Registration order is itself deterministic (board order at fight start, then the deterministic fire order each step), so the sweep is bit-reproducible. Simplest thing that works â€” resolves the Combat-manager *within-step component order* open. *(Phase 1 realization, 2026-06-05: the sweep is built as fixed type-ordered passes â€” item cooldowns â†’ statuses â†’ Delivery travel, each in insertion order â€” which delivers the same deterministic, bit-reproducible order. A literal `seq_id` counter field is deferred until components of different types must interleave by registration time, which Phase 1 never requires.)*
25. **Scene tree = a logic tree (under the `Game` autoload) + a separate presentation tree (the main scene).** Combat logic is plain `RefCounted` (`Actor` / `Item` / `Status` / `Delivery` / `Timekeeper` / event bus); only the three orchestrators (`Run manager` / `Encounter` / `Combat manager`) are `Node`s, and **only the `Combat manager` runs `_physics_process`** (the one tick). The presentation tree (`main.tscn` â†’ `ScreenHolder` â†’ title/run/death screens; the run screen composing the corridor panel + a swappable `CombatView` + overlays) only *reads* the logic and emits intents â€” so a headless autotest just doesn't mount it. The **framed-vs-full-screen** open is isolated to the single swappable `CombatView` sub-scene. Full structure + directory layout: [`architecture.md` â†’ Scene tree & node model](architecture.md#scene-tree--node-model).
